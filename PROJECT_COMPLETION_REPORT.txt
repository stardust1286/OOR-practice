【第三部分】實現的 OOP 特性 ✅
═══════════════════════════════════════════════════════════════════════════════

【1】封裝 (Encapsulation) ✅
┌─────────────────────────────────────────────────────────────┐
│ • struct Vehicle 封裝車輛狀態和方法                          │
│ • Car, Bus, Truck 各有專屬欄位                              │
│ • .c 檔隱藏實現細節，.h 檔暴露公開介面                      │
│ • private static 變數維護控制器私有狀態                     │
│                                                             │
│ 示例：                                                       │
│   parking_spaces[] → static 陣列，外部無法直接訪問         │
│   parking_revenue → static 變數，私有統計                  │
└─────────────────────────────────────────────────────────────┘

【2】繼承 (Inheritance) ✅
┌─────────────────────────────────────────────────────────────┐
│ • 所有車型包含 Vehicle 基類欄位                              │
│ • 子類新增自己的特有欄位                                    │
│ • 子類透過巨集覆蓋虛擬方法指標                              │
│                                                             │
│ 類層次結構：                                                │
│   Vehicle (基類)                                            │
│   ├─ Car (轎車)                                             │
│   ├─ Bus (公車)                                             │
│   └─ Truck (卡車)                                           │
└─────────────────────────────────────────────────────────────┘

【3】多型 (Polymorphism) ✅
┌─────────────────────────────────────────────────────────────┐
│ • 虛擬方法表：vehicle->move, ->stop, ->park               │
│ • 同一方法呼叫，三種車執行不同邏輯                          │
│ • 在執行時根據型別動態決定行為                              │
│                                                             │
│ 多型示例：                                                  │
│   car1.base.move(&car1, 10)   → 執行 car_move()            │
│   bus1.base.move(&bus1, 10)   → 執行 bus_move()            │
│   truck1.base.move(&truck1, 10) → 執行 truck_move()        │
│   ≠ 三種完全不同的邏輯！                                    │
│                                                             │
│ 控制器多型：                                                │
│   signal_ctrl.control_vehicle() → 紅綠燈決策               │
│   parking_ctrl.control_vehicle() → 停車場決策              │
│   ≠ 完全不同的決策邏輯                                     │
└─────────────────────────────────────────────────────────────┘

【4】介面 (Interface) ✅
┌─────────────────────────────────────────────────────────────┐
│ • ITrafficController 定義通用介面                            │
│ • 所有控制器遵循同一簽名                                    │
│ • 新增控制器無需修改舊代碼                                  │
│ • 實現開閉原則 (Open-Closed Principle)                     │
└─────────────────────────────────────────────────────────────┘

【虛擬方法多型驗證】

scenario: 三輛車各行駛 10 km

  Car 101 move(10 km)
  ├─ 執行：car_move(&car1, 10)
  ├─ 油耗：10 * 0.08 = 0.8 L
  ├─ 位置：x += 7.0, y += 3.0
  └─ 速度：60 km/h (轎車特性)

  Bus 201 move(8 km)
  ├─ 執行：bus_move(&bus1, 8)
  ├─ 油耗：0 L (公車用電)
  ├─ 位置：x += 4.0, y += 4.0
  └─ 速度：40 km/h (受乘客數影響)

  Truck 301 move(15 km)
  ├─ 執行：truck_move(&truck1, 15)
  ├─ 油耗：15 * 0.15 = 2.25 L (重型車耗油多)
  ├─ 位置：x += 9.0, y += 6.0
  └─ 速度：50 km/h (正常巡航)

✓ 三個相同的呼叫簽名 vehicle->move()
✓ 執行時動態決定執行哪個函式
✓ 結果完全不同，展現多型威力


【控制器多型驗證】

scenario: 三輛車分別通過紅綠燈

  Signal Controller:
  ├─ Car 101: GREEN → SUCCESS
  ├─ Bus 201: GREEN + PUBLIC_PRIORITY → SUCCESS
  └─ Truck 301: GREEN but PEAK_TIME → DENIED

  Parking Controller:
  ├─ Car 101: FOUND_SPACE + CHARGE_5 → SUCCESS
  ├─ Bus 201: FOUND_SPACE + FREE → SUCCESS
  └─ Truck 301: FOUND_SPACE + CHARGE_20 → SUCCESS

✓ 同一方法簽名 control_vehicle()
✓ 不同控制器，完全不同邏輯
✓ 體現介面驅動設計

【第五部分】設計決策與理由 ✅
═══════════════════════════════════════════════════════════════════════════════

【決策 1】為什麼用 struct 嵌入而不是指標繼承？
┌─────────────────────────────────────────────────────────────┐
│ struct Car {                                                │
│     Vehicle base;  ← 嵌入，不是 Vehicle *base;             │
│     int fuel;                                               │
│ }                                                           │
│                                                             │
│ 理由：                                                       │
│ ✓ 棧分配，無需動態記憶體                                    │
│ ✓ 自動生命週期管理                                          │
│ ✓ 訪問效率高                                                │
│ ✓ 韌體應用的標準做法                                        │
│ ✓ 對應 PIDAW 等教學案例的模式                              │
└─────────────────────────────────────────────────────────────┘

【決策 2】為什麼虛擬方法是 void *self？
┌─────────────────────────────────────────────────────────────┐
│ typedef void (*funtypeVehicle_Move)(void *self, float d);   │
│                                                             │
│ 理由：                                                       │
│ ✓ 允許任何型別的物件傳入                                    │
│ ✓ 實現真正的通用介面                                        │
│ ✓ 內部透過強制轉型獲得具體型別                              │
│ ✓ C 語言模擬多型的標準模式                                  │
└─────────────────────────────────────────────────────────────┘

【決策 3】為什麼用巨集初始化而不是函式？
┌─────────────────────────────────────────────────────────────┐
│ #define CAR_DEFAULTS { \                                    │
│     .base = { \                                             │
│         ...,\                                               │
│         (funtypeVehicle_Move)car_move, \  ← 自動綁定     │
│     }, \                                                    │
│     0, MAX_FUEL  \                                          │
│ }                                                           │
│                                                             │
│ Car myCar = CAR_DEFAULTS;  ← 棧分配 + 自動初始化           │
│                                                             │
│ 理由：                                                       │
│ ✓ 編譯時完全初始化                                          │
│ ✓ 函式指標自動綁定                                          │
│ ✓ 零額外開銷                                                │
│ ✓ 對應 PIDAW_DEFAULTS 模式                                 │
└─────────────────────────────────────────────────────────────┘

【決策 4】為什麼要 private static 控制器狀態？
┌─────────────────────────────────────────────────────────────┐
│ // signal_controller.c 內部                                │
│ static uint8_t current_light_state = 0;   ← private        │
│ static uint32_t signal_cycle = 0;                           │
│ static uint32_t total_passed = 0;                           │
│                                                             │
│ 理由：                                                       │
│ ✓ 狀態只在本檔案可見                                        │
│ ✓ 防止外部直接修改                                          │
│ ✓ 模擬 C++ 的 private 成員                                 │
│ ✓ 符合封裝原則                                              │
└─────────────────────────────────────────────────────────────┘

【決策 5】為什麼每個車型的 move() 邏輯完全不同？
┌─────────────────────────────────────────────────────────────┐
│ car_move():    oil -= distance * 0.08  (省油)              │
│ bus_move():    速度 = 40 * (1 - passenger_ratio)  (考乘客)  │
│ truck_move():  複雜計算，考慮載重影響                        │
│                                                             │
│ 理由：                                                       │
│ ✓ 展示真實物理模型                                          │
│ ✓ 體現多型的價值                                            │
│ ✓ 同一方法簽名，三種行為 = 多型精髓                        │
│ ✓ 教學效果最大化                                            │
└─────────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════════════════════

【教學文檔概念】→ 【本項目實現】

1. Struct 當作 Class
   ────────────────────────────────────────────
   PIDAW struct 定義          → Vehicle struct 定義
   包含資料 + 函式指標         → 同樣包含狀態 + 虛擬方法

2. 函式指標 = 虛擬方法
   ────────────────────────────────────────────
   funtypePIDAW_Calc          → funtypeVehicle_Move
   funtypePIDAW_Reset         → funtypeVehicle_Stop
   使用：myPid.calc(&myPid)   → 使用：myCar.base.move(&myCar)

3. 巨集初始化 = 構造函數
   ────────────────────────────────────────────
   #define PIDAW_DEFAULTS     → #define CAR_DEFAULTS
   自動初始化和綁定函式       → 同樣自動初始化和綁定

4. 介面規格 (多型)
   ────────────────────────────────────────────
   IEncoder 介面              → ITrafficController 介面
   TamagawaEncoder           → SignalController
   SyntecEncoder             → ParkingController
   同一介面，多個實現         → 同一介面，多個控制器

5. 資料流與狀態
   ────────────────────────────────────────────
   編碼器 → PID → 控制 → 馬達  → Vehicle → Controller → 決策
   單向鏈式處理資料           → 同樣單向資料流
